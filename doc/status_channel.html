Status information

During the runtime, many events may occur. This section describes how the events are passed to user application and which kind of events are available.

Status channel is used for events with debugging and informational character. Usage of status channel is optional, user can select which events to subscribe.

For critical exceptions, an interrupt is used. Receiving of the interrupt usually means that something goes wrong and the application should be stopped.

The API can print status messages or store them into file, including PCAP record of order sender communication with market.

Status channel

Subscription to status channel can be done using api.statusChannel.subscribe() with an argument containing or’ed list of desired events.

For reading the events, two methods are provided: api.statusChannel.getEvent() blocks the user application until an event is available in the queue, api.statusChannel.getEventNonBlock() is a non-blocking version.

Event code Description
EVT_ORDER_INVALID_VALUE Invalid value of parameter on order sender input. The event structure contains information which parameters are wrong (Time in force, order type, …).
EVT_ORDER_WINDOW_FULL Order couldn’t be sent due to TCP flow control mechanism.
EVT_ORDER_FIFO_FULL Order couldn’t be sent - FIFO full.
EVT_ORDER_ALL Subscribe all order sender events.
EVT_RISK_CHECK Pre-trade risk check unit dropped an order request. The event structure contains information which parameter exceeded configured limit.
EVT_BOOK_UNKNOWN_SYMBOL The book received symbol not present in mapping file.
EVT_BOOK_AUTOMATIC_RESET Reset of Full Order Book finished
EVT_BOOK_ALL Subscribe all book events
EVT_ARBITER_SEQ_HOLE The arbiter has detected missing packets in the market stream.
EVT_DECODER_BAD_PACKET Decoder received broken packet - skipping.
EVT_RECOVERY_STATUS Recovery status change (finished or failed).
EVT_RECOVERY_DONE Recovery finish and fail reports
EVT_ALL Subscribe all events.

Interrupt

Interrupts are non-maskable, their presence indicates a serious problem. Please, contact Magmio a.s. if you are experiencing issues with interrupts. Below is an example of interrupt notification:

WARNING: Interrupt received at Mon Jun 10 12:53:43 2019
ERROR: Order sender: Watchdog expired, order sender in the card was killed

Logs

To modify verbosity of the information printed to standard error output, the user can use Log class method:

void Log::setPrintVerbosity(const tVerbosityLevel verbosity);

Following verbosity levels are available:

  • QUIET - No messages are displayed
  • ERROR - Error messages only are displayed
  • WARNING - Error and warning messages are displayed
  • INFO - Basic information messages are displayed
  • DEBUG - Detailed information messages are displayed

Magmio can also store the log messages into file and create pcap file containing recorded communication with market (TCP) or multicast (IGMP). To do so, call createFile function to set log verbosity level and open logging file, and use createOrderPcap to create PCAP log of order data or createMulticastPcap to create PCAP log of multicast data.

TError Log::createFile(const tVerbosityLevel verbosity, const char *path);
TError Log::createOrderPcap(const char *path);
TError Log::createMulticastPcap(const char *path);

It is also possible to record entire traffic using record method, however this method only allows to record traffic from single port (A), which is configured in hardware.

TError PcapReader::record(std::string filename);

Please, note that the pcap logging should be used for debugging purposes only. It adds additional overhead to packet processing, thus it may have negative impact on performance.
