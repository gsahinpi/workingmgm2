Magmio User's Manual

Order Sender

Order Sender is instantiated as a part of Magmio API and cooperates with low-latency, order sending component in the card. It maintains session to market order entry gateway and allows a user to control sending of orders from the card. Multiple sessions are allowed, the number of supported sessions can be modified if needed.

Order Sender configuration

Market session

Each session is uniquely identified by session ID. The value can be configured in API configuration file using id attribute of session tag. In the API the session ID can be passed to certain methods to refer only to specific session. Also, all events contain session ID of the session, from which the event was originated.

To be able to successfully connect to desired market order gateway, user has to choose proper order entry protocol. This is done using tag <protocol type="">. Following table shows list of all supported order protocols, concrete list depends on Magmio package version.

To establish a market session, several fields are required, e.g. user login and password. These parameters are taken (together with network configuration) from the API configuration file of the specified order protocol.

Instrument identification

Instrument which will be traded is usually described using string called symbol (i.e. “AAPL”) and some additional fields (i.e. strike price, strike date for options). The fields are stored in mapping file, right after symbol ID field and group ID field, and automatically passed to Order Sender in the card.

39915;1;BRBY;20121222;270000;C;100;4
39916;1;AAPL;20121222;280000;C;100;4
39917;1;CSCO;20140118;30000;P;100;4

Order parameters

Each order message generated in the card consists of price and size of the order and another parameters setting behaviour of the order. These parameters are configurable via user written trading logic (order sender is connected to trading logic output).

List of available parameters:

  • ID of generated order *
  • Price of order
  • Size of order
  • Minimum quantity to trade
  • Message type *
  • Order side *
  • Order type*
  • Position *
  • Time in force *
  • Display *
  • ISO *

Certain markets do not support some of listed parameters or its values. In that case, the parameter is ignored.

Parameters marked with * are checked for correct value at the order sender input. If an invalid configuration is detected, no order is generated. Instead, EVT_ORDER_INVALID_VALUE message is send to the status channel and “Bad Orders” statistic counter is incremented.

Order Sender management

Each instance of Order Sender in Magmio API provides several methods for monitoring and controlling order generator in the card and for session management. Following text describes basic functionality of Order Sender. For more details, refer to the Order Sender example in example directory.

Managing a session

After Magmio API is initialized, Order Sender is by default disconnected from market. When connecting, Order Sender connects to the market server using settings from the configuration file. To connect to the market, use one of the following methods

  • api.order.start()
  • api.order.start("session_id")

If the session ID is not specified, Order Sender connects to all sessions with automap="true" specified in the configuration file. If the session ID is specified, Order Sender searches the configuration file for a session with specified id and automap="true" and will enable just this session.

Once connected, Order Sender automatically handles some market protocol events without user interaction (such as responding to heartbeat requests). Use api.order.stop() to disconnect from market and stop Order Sender(s).

Enable Order Sender

To enable Order Sender to generate orders to market, use method api.order.enable(). Besides the session ID, the method also takes optional argument saying how many orders (default 1) the order sender can send before it disables itself. If set to 0 (unlimited number of orders), the Order Sender stays enabled until it is manually disabled by user.

Order Sender can be disabled any time by calling api.order.disable().

Watchdog

We recommend using watchdog timer to prevent uncontrolled sending of orders to market when the user application freezes or dies unexpectedly.

The watchdog is enabled by calling api.order.wdt_start() with argument configuring the watchdog timeout in milliseconds. The user application then has to call api.order.wdt_reset() periodically to reset the timer.

If the reset doesn’t come in configured time, i.e. because the application has frozen, the timer expires and all connections in the card are closed independently on the software state.

Watchdog can be disabled by api.order.wdt_stop().

Software Orders

It is possible to generate orders directly via software using API methods sendOrder / sendOrderFpga, see Doxygen for more information.

  • api.order.sendOrderFpga() - Sends order directly to the Order Sender in FPGA (same as for orders generated directly by user strategy)
  • api.order.sendOrder() - Software order (send orders purely through software)

Note

The orders generated by calling api.order.sendOrder() in host application ignore Risk Check module and are excluded from the statistic counters for sent orders from the FPGA card.

Reading events

When an order is sent from card to the market server, an event is generated. Events are also generated when market server responds to requests from Order Sender (order acknowledgement, rejection, cancellation, execution).

The following method is provided for reading the order events:

  • order.getEvent() - Blocks the application execution until a new event is available in the queue.

Each event contains:

  • Event code
  • Session ID (as defined by user in the config file) of session from which this event originated
  • Type of protocol used in this session (Nasdaq OTTO, CME iLink,…)
  • Precise timestamp from FPGA upon packet arrival
  • Latency of shot order / latency between last two events
  • Information from which physical port on the card the order was shot
  • Text field populated in case of error/rejection
  • Structure with order information

Definition of the structure and event codes is available in order_types.h. Following event codes are available. Please, note that some order protocols may not use all of the listed events.

Event code Description
EVT_ORDER_SEND Order from FPGA has been sent to market (size and price contain requested values)
EVT_CANCEL_SEND Cancel request from FPGA has been sent to market
EVT_REPLACE_SEND Replace request from FPGA has been sent to market (size and price contain new values)
EVT_ORDER_ACK Market acknowledged the order request (size and price contain accepted values)
EVT_ORDER_REPLACED Market acknowledged the order replace request / order was modified
EVT_PRIORITY_CHANGE Priority of the order has been changed in the system
EVT_CANCEL_ACK Request for cancelling acknowledged/pending
EVT_ORDER_FILLED Order has been (partially) filled (size contains quantity executed by this event, not accumulated value)
EVT_ORDER_KILLED Order has been canceled/killed (size contains quantity decremented from the order by this event, not accumulated value)
EVT_ORDER_KILLED_AIQ Order was cancelled due to broken AIQ (self-match prevention)
EVT_DONE_FOR_DAY No further executions forthcoming for the trading day (size contains total traded quantity)
EVT_MESSAGE_REJECTED Message to market was rejected, see info_text field for reason. Order structure may not be populated
EVT_LATENCY_MEASURED Connection latency was measured, timestamp contains measured latency
EVT_SYSTEM_EVENT System event message from market (i.e. start/end of the market day)
EVT_BROKEN_TRADE Trade was broken.
EVT_REPLAY_COMPLETE When data recovery is complete and session is ready to enable order sender in HW
EVT_ERROR Unrecoverable error occurred, order sender has been stopped. See info_text field for error description
EVT_STOP When session is gracefully stopped, this is the last event for a session

The information about order is represented by protocol-independent structure and contains following items. Exact type definition is available in order_types.h.

uint64_t orderID;              // API internal order ID. Use it for canceling order in cancel method
uint64_t newOrderId;           // Populated for replace order messages.
uint64_t marketId;             // Order ID as assigned by market - binary number
std::string marketIDStr;       // Order ID as string when binary representation is not possible (i.e. CME FIX)
uint32_t sequenceNumber;       // Sequence number of this message from underlying protocol (if available)
uint64_t execID;               // Execution ID / Match Number (if available)
uint64_t marketTimestamp;      // Message timestamp assigned by market (if available)
uint64_t sid;                  // ID of instrument which caused sending of order to market
uint8_t userTag;               // User assigned tag to order in strategy

int64_t price;                 // Price in scale of price_scale field.
int8_t  priceScale;            // Price scale 4 means that price 10000 = 1$
uint32_t size;                 // Order QTY. Contains requested quantity in ACK message, increment quantity in FILL message, decrement qty in CANCEL message
uint32_t minimumQty;           // Minimum quantity needed to execute this order. Not applicable on some markets.

TOrderSide side;               // Buy/sell side
TOrderPosition position;       // Open/close position. Not applicable on some markets
TOrderTIF timeInForce;         // Time in force
TOrderType orderType;          // Type of order
TDisplay displayType;          // Not applicable on some markets
TIso iso;                      // Not applicable on some markets

std::string symbol;            // Security symbol. Used when the symbol can't be represented as a number (field symbolNum)
uint64_t symbolNum;            // Security symbol as number / token (if possible)
uint32_t strikePrice;          // Strike price. Not applicable on some markets
std::string strikeDate;        // Strike date. Not applicable on some markets
TOrderPutCall putCall;         // Option put/call. Not applicable on some markets

char systemEventCode;          // Set when EVT_SYSTEM_EVENT message. Not applicable on some markets

uint8_t extras[EXTRAS_WIDTH];  // Extra protocol specific items. Refer to msgs/<proto_name>/order_extras.h for more info

In case the API gets market response to order which no longer exists, the structure itself may not be populated (just orderId). Please, note that price and size fields in the order structure are changing during lifespan of order:

  1. EVT_ORDER_SEND - requested price and size
  2. EVT_ORDER_ACK - price and size accepted by market
  3. EVT_ORDER_FILLED - size (increment, not accumulated value) and price of this fill
  4. EVT_ORDER_KILLED - size contains killed quantity by this kill/cancel event, not accumulated value

Note

The structure is unified between different market protocols. It may happen that certain markets do not populate some of the listed items. In that case, the unavailable items will be set to default/zero value.

Order extra fields in events

Some order protocols in Magmio support additional extra fields to be sent from FPGA to the software application included in each order event. Declaration of the structure representing extra fields for a specific protocol is available in header file: /usr/include/magmio/msgs/<protocol>/order_extras.h.

The following code snippet provides an example of catching order sender events and parsing available fields including the extra fields. The given code should be used in the software application and/or software model, for a full example see order_example.cpp source file provided as a part of the software model. The short explanation of the following code is given in comments.

// Import header file with extra fields declaration for the specific order protocol
#include <magmio/msgs/bist_ouch/order_extras.h>

...

// Almost infinite loop for processing received order events. The loop should be executed in a separate thread to not block the main application process
while (!stop) {
   Magmio::TEventItem event;
   // Pointer to the extras structure for the specific order protocol
   Magmio::TBistExtras* extras;

   // 1000 = wait for an event for 1000 ms
   //   -1 = wait for an event infinitely
   if (!api.order.getEvent(event, 1000)) {
      // false is returned when there is no event to get
      continue;
   }

   std::cout << "Got event from session: " << event.sessionID << std::endl;

   // Process received events according to their type
   switch (event.type) {
      case Magmio::TEventType::EVT_ORDER_SEND:
         std::cout << "New order with orderToken = " << event.order.orderID  << " sent." << std::endl;
      break;

      case Magmio::TEventType::EVT_ORDER_ACK:
          std::cout << "Order with orderToken = " << event.order.orderID << " ACKED (orderID = " << event.order.marketID <<  ")" << std::endl;
          // Cast the extras event order member to the specific extras structure according to the order protocol
          extras = (Magmio::TBistExtras*)event.order.extras;
          // Access individual extras fields
          std::cout << "Order state: " << extras->orderState; << std::endl;
      break;

      case Magmio::TEventType::EVT_ORDER_FILLED:
          std::cout << "Order with orderToken = " << event.order.orderID << " FILLED (trade QTY = " << event.order.size <<  ")" << std::endl;
          extras = (Magmio::TBistExtras*)event.order.extras;
          // Here, the matchID is available via extras->matchID as an array of 12B
      break;

      case Magmio::TEventType::EVT_ORDER_REPLACED:
          std::cout << "Order with orderToken = " << event.order.orderID << " REPLACED (replacementOrderToken = " << event.order.newOrderID <<  ")" << std::endl;
          // Further messages to replaced order (i.e. fills, kills) will be reported with event.order.orderID set to value of replacement orderToken
      break;

      ...
      // Add case block for all event types you want to process
   }
   ...
}
